═══════════════════════════════════════════════════════════════════════════
                    ERROR HANDLING FLOW
═══════════════════════════════════════════════════════════════════════════

┌─────────────────────────────────────────────────────────────────────────┐
│                       ERROR HANDLING LAYERS                              │
└─────────────────────────────────────────────────────────────────────────┘

LAYER 1: INPUT VALIDATION (FastAPI)
────────────────────────────────────
User uploads text file
  │
  ├─> FastAPI validates content-type
  │   └─> if not image/*:
  │       └─> HTTPException(400, "Invalid file type")
  │           └─> User sees: {"detail": "Invalid file type..."}
  │
  └─> Early exit, no processing

LAYER 2: FILE VALIDATION (Orchestrator)
────────────────────────────────────────
FastAPI calls orchestrator.analyze_screenshot("/nonexistent.png")
  │
  ├─> Orchestrator checks Path(image_path).exists()
  │   └─> if not exists:
  │       └─> FileNotFoundError("Image not found: /nonexistent.png")
  │           └─> FastAPI catches, returns HTTP 500
  │           └─> MLFlow logs: success=0, error="Image not found"
  │
  └─> Early exit, no VLM call

LAYER 3: VLM ERRORS (Orchestrator)
───────────────────────────────────
VLM call fails (timeout, connection refused, etc.)
  │
  ├─> agent.run() raises Exception
  │   └─> Orchestrator catches in try/except
  │       ├─> logger.error("Analysis failed: {e}")
  │       ├─> MLFlow logs: success=0, error=str(e)
  │       └─> raise ValueError("VLM analysis failed: {e}")
  │           └─> FastAPI catches, returns HTTP 500
  │
  └─> User sees clear error message

LAYER 4: TOOL ERRORS (Tool.forward)
────────────────────────────────────
Tool execution fails (OpenCV error, OCR failure, etc.)
  │
  ├─> Tool.forward() raises Exception
  │   ├─> Tool logs: logger.error("Tool execution failed: {e}")
  │   └─> Re-raise to agent
  │       └─> Agent propagates to orchestrator
  │           └─> Orchestrator catches (Layer 3)
  │               └─> MLFlow logs error
  │
  └─> Comprehensive error context

LAYER 5: CLEANUP (FastAPI - finally block)
───────────────────────────────────────────
Always runs, even if error occurred
  │
  └─> finally:
      └─> Path(tmp_path).unlink(missing_ok=True)
          └─> Temp file deleted regardless of success/failure
          └─> Prevents disk space leaks

═══════════════════════════════════════════════════════════════════════════
                    SCALING STRATEGY
═══════════════════════════════════════════════════════════════════════════

FASTAPI PODS (CPU-bound):
  - HorizontalPodAutoscaler
  - Scale on: CPU utilization > 70% or requests/pod > 50
  - Min replicas: 1
  - Max replicas: 10
  - Scale-up: Add 1 pod every 30s
  - Scale-down: Remove 1 pod every 5min

VLLM PODS (GPU-bound):
  - Manual scaling (GPU resources expensive)
  - Scale on: GPU utilization > 80% or request queue > 10
  - Min replicas: 1
  - Max replicas: 3 (limited by GPU availability)

SCALING EXAMPLE:

Traffic spike: 10 req/s → 100 req/s

  FastAPI:
    0min:  3 pods handle 10 req/s (low CPU ~30%)
    1min:  Traffic increases, CPU → 80%
    2min:  HPA adds 2 more pods (now 5 pods)
    3min:  CPU drops to 50%, stable
    10min: Traffic drops to 20 req/s
    15min: HPA removes 1 pod (now 4 pods)
    20min: Removes another (now 3 pods)

  vLLM:
    - 1 pod handles all traffic (GPU efficient with batching)
    - Queue builds up if >30 concurrent requests
    - Manual intervention to add pod if sustained high load


═══════════════════════════════════════════════════════════════════════════
                    DISASTER RECOVERY
═══════════════════════════════════════════════════════════════════════════

BACKUP STRATEGY:

Daily:
  - MLFlow data (mlruns/) → S3
  - Model checkpoints → S3
  - Application config → Git

Weekly:
  - Full database backup (if using DB for feature store)

RECOVERY TIME OBJECTIVES (RTO):

Component failure:
  - FastAPI pod crash: <30s (K8s auto-restart)
  - vLLM pod crash: <2min (model load time)
  - vLLM server hung: <5min (health check + restart)

Data loss:
  - MLFlow data: <24h (restore from daily backup)
  - Model weights: <1h (re-download from HuggingFace or S3)

Complete cluster failure:
  - RTO: <4h (provision new cluster, deploy from Git, restore data)
  - RPO: <24h (daily backups)


═══════════════════════════════════════════════════════════════════════════
                    KEY TAKEAWAYS
═══════════════════════════════════════════════════════════════════════════

1. ERROR HANDLING IS MULTI-LAYERED
   - Input validation (FastAPI)
   - Business logic validation (Orchestrator)
   - Tool execution errors (Tools)
   - Always cleanup resources (finally blocks)

2. OBSERVABILITY IS ESSENTIAL
   - Logging: Structured logs with context
   - Metrics: Success rate, latency, throughput
   - Tracing: MLFlow for experiment tracking
   - Alerting: Proactive issue detection

3. PRODUCTION DEPLOYMENT IS DIFFERENT
   - Multiple replicas for high availability
   - Auto-scaling based on load
   - Health checks for self-healing
   - Monitoring and alerting

4. SEPARATION OF CONCERNS
   - FastAPI: Web layer (HTTP, file upload)
   - Orchestrator: Business logic (validation, orchestration)
   - Smolagents: Agent framework (tool dispatch)
   - Tools: Domain logic (extraction)

5. FAIL FAST, LOG EVERYTHING
   - Validate early (don't waste VLM time)
   - Log errors with context
   - Return clear error messages
   - Track everything in MLFlow

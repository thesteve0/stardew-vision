═══════════════════════════════════════════════════════════════════════════
                        REQUEST FLOW (DETAILED)
═══════════════════════════════════════════════════════════════════════════

1. CLIENT SENDS REQUEST
   ────────────────────
   POST http://localhost:8001/v1/chat/completions
   {
     "model": "Qwen2.5-VL-7B-Instruct",
     "messages": [...],
     "tools": [...],
     "temperature": 0.7,
     "max_tokens": 500
   }

2. HTTP SERVER VALIDATES
   ─────────────────────
   - Check model exists
   - Validate JSON schema
   - Check required fields
   - Enqueue request

3. REQUEST QUEUE
   ─────────────
   Waiting requests: [Request A, Request B, Request C]
   Engine pulls batch: [Request A, Request B]  ← Dynamic batching!

4. VLLM ENGINE PROCESSES BATCH
   ───────────────────────────
   PagedAttention:
     Request A: Needs 256 KV cache blocks
     Request B: Needs 128 KV cache blocks
     Allocate blocks efficiently (no fragmentation)

   Tokenization:
     Request A: [1, 234, 567, ...]  (prompt tokens)
     Request B: [1, 890, 123, ...]

5. MODEL EXECUTOR RUNS
   ───────────────────
   Forward pass on GPU:
     - Vision encoder processes image (if present)
     - Language model processes tokens
     - Returns logits

   Sampling:
     - Apply temperature
     - Apply top_p/top_k
     - Sample next token

   Repeat until:
     - max_tokens reached
     - <EOS> token generated
     - Stop sequence matched

6. RESPONSE FORMATTING
   ───────────────────
   Generated tokens: [7890, 1234, 5678, ..., <EOS>]
   Decode: "Here's the extracted data..."

   Format as OpenAI response:
   {
     "id": "chatcmpl-abc123",
     "object": "chat.completion",
     "choices": [{
       "message": {
         "role": "assistant",
         "content": "Here's the extracted data...",
         "tool_calls": [...]
       }
     }]
   }

7. CLIENT RECEIVES RESPONSE
   ────────────────────────
   HTTP 200 OK
   {JSON response}


═══════════════════════════════════════════════════════════════════════════
                        PAGEDATTENTION EXPLAINED
═══════════════════════════════════════════════════════════════════════════

NAIVE ATTENTION (Transformers):
  - Allocates contiguous memory for KV cache
  - Wastes memory due to fragmentation
  - Can't batch efficiently

  Request A: [████████████████                ]  Allocated: 100 tokens
  Request B: [████████                        ]  Allocated: 100 tokens
                         ^^^^^^^^^^^^^^^^^^^^^^^^^  Wasted!

PAGEDATTENTION (vLLM):
  - Allocates memory in blocks (like OS virtual memory)
  - No fragmentation
  - Can batch requests of different lengths

  Block Pool: [█][█][█][█][█][█][█][█][█][█]
               ↑           ↑
  Request A: Uses 4 blocks
  Request B: Uses 2 blocks
  Request C: Uses 3 blocks
  Total: 9 blocks used, 1 free (90% utilization!)

PERFORMANCE GAIN: 2-4x higher throughput


═══════════════════════════════════════════════════════════════════════════
                        CONTINUOUS BATCHING
═══════════════════════════════════════════════════════════════════════════

STATIC BATCHING (Naive):
  Batch 1: [Request A, Request B]
  Wait for both to complete (slowest request determines latency)

  Request A: ████████ (8 tokens, done)
  Request B: ████████████████████ (20 tokens)
               ↑ A waits here even though it's done!

  Batch 2: [Request C, Request D] ← Can't start until Batch 1 done

CONTINUOUS BATCHING (vLLM):
  When Request A finishes, immediately add Request C to batch

  Step 1-8:  [Request A, Request B]
  Step 9-20: [Request B, Request C] ← A replaced by C!
  Step 21-30:[Request B, Request C, Request D]

  No waiting! GPU always at max utilization.

═══════════════════════════════════════════════════════════════════════════
                        CONFIGURATION OPTIONS
═══════════════════════════════════════════════════════════════════════════

vllm serve MODEL_PATH [OPTIONS]

COMMON OPTIONS:

--port 8001
  HTTP server port

--dtype float16
  Model precision (float16, bfloat16, float32)
  ⚠️ ROCm 7.2 only supports float16!

--enable-tool-calling
  Enable function calling support

--max-model-len 4096
  Maximum context length (saves GPU memory)

--gpu-memory-utilization 0.9
  GPU memory to allocate (0.0-1.0)

--max-num-seqs 256
  Maximum number of sequences in batch

--tensor-parallel-size 1
  Number of GPUs for tensor parallelism

--trust-remote-code
  Allow model code from HuggingFace (required for some models)

═══════════════════════════════════════════════════════════════════════════
                        PERFORMANCE METRICS
═══════════════════════════════════════════════════════════════════════════

THROUGHPUT:
  Requests/second: Depends on:
    - Request batch size
    - Sequence length
    - Model size
    - GPU memory

  Example (Qwen2.5-VL-7B on AMD Strix Halo):
    Single request: ~10 tokens/second
    Batch of 4: ~35 tokens/second (3.5x throughput!)

LATENCY:
  Time to first token (TTFT):
    - Prefill phase (process prompt)
    - ~200-500ms for typical prompt

  Time per output token (TPOT):
    - Decode phase (generate tokens)
    - ~100ms per token

  Total latency = TTFT + (num_tokens × TPOT)
  Example: 500ms + (20 tokens × 100ms) = 2.5 seconds

MEMORY:
  GPU memory usage:
    - Model weights: ~14 GB (FP16 for 7B model)
    - KV cache: ~2 GB per 1000 tokens
    - Activation memory: ~2 GB

  Total for Qwen2.5-VL-7B: ~18-20 GB
  Fits on: AMD Strix Halo (16 GB shared), NVIDIA A100 (40 GB)
